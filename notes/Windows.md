几个原则：

# KISS原则
Keep It Simple Stupid，保持简单愚蠢。复杂的东西容易出错，难以维护，除非清楚的知道自己在做什么并且确定利大于弊。

# 最小权限原则
只安装和使用必要的软件和服务，只给最低限度的提权，减少系统的攻击面和维护成本。


几个概念：
- 提权：提升权限，比如从普通用户提升到管理员用户。
- 沙盒：隔离环境，比如用虚拟机或者容器运行不信任的软件，防止其影响主系统。

如果是开箱即用的品牌机，那就开箱即用，无需折腾分区（KISS原则）

# 关于软件安装
软件按照这么几个维度分类：
- 是否必须安装，还是双击.exe文件就能用？
- 是否经常使用，还是偶尔使用？
- 是否非用不可？
- 是否有劣迹历史？

选软件的一点经验：
自己写的 > 开源的 > 闭源的
Portable 的 > 必须安装的
别人花钱买的 > 免费的 > 需要自己付费的
行业通用的 > 小众的
口碑好的 >>> 有黑历史但不得不用的

偶尔用的软件和有黑历史的软件，放在沙盒里面用，或者放在二奶电脑上用。

至于组织形式，可以是 C:/opt/公司名/软件名/版本号/软件本身的目录结构
如果软件不需要保留多个版本，那就省略版本号这级
如果软件的名下只有这么一款软件，那也可以省略公司名这级

绿色版的办法包括：
重装/重置 之前将已经安装的软件从安装目录下的软件目录拷贝出来，比如放在C:/opt/目录下，或者放进移动硬盘。opt的命名是跟着Linux来的。在形成自己的有效的命名规范之前，沿袭行业规范是个不错的选择。例如 https://www.runoob.com/linux/linux-system-contents.html

绿色的软件要找到能启动的那个 exe 文件，右键创建快捷方式放到开始菜单。办法可以问 chatGPT
有时候软件会依赖 vcredist 等运行库，首次运行的时候会提示找不到 dll 文件，
- 可以去下载对应的 vcredist 安装，
- 可以解压 vcredist 将 dll 放在软件目录下
- 可以去之前备份的文件系统里面或者别人的电脑上搜对应的 dll 放到软件目录下

# 如何做配置
用过 VS Code 的一定对 VS Code 的配置风格印象深刻：
- Machine 级别的配置，配置了会影响机器上的所有用户
- User 级别的配置，配置了会影响当前用户的所有项目
- Workspace 级别的配置，配置了只会影响当前项目，无论在哪台机器上打开这个项目都会生效，而且会覆盖 User 和 Machine 级别的配置

所以在做配置的时候要有这样的意识：
- 机器级别的配置，尽量少做，除非是必须的
- 用户级别的配置，可以多做
- 项目级别的配置，尽量多做

在排查配置错误的时候也是一样的思路，从项目级别开始排查，然后是用户级别，最后是机器级别

例如 Python 的环境变量配置，给每个项目都创建一个 venv，对应应项目的 Python 解释器和依赖包，这样就不会污染用户级别和机器级别的 Python 环境
